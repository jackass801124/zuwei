# 紫微斗數 Gemini 1.5 Pro 實施指南\n\n## 概述\n\n本指南提供完整的 Gemini 1.5 Pro API 整合實施步驟。系統已遷移至 Google Gemini，實現多模型混合策略、上下文快取、多模態輸入、結構化 JSON 輸出等高級功能。\n\n## 核心特性\n\n### 多模型混合策略\n\n系統採用混合模型方案以平衡成本與性能：\n\n| 模型 | 用途 | Token 限制 | 成本 | 特性 |\n|------|------|----------|------|------|\n| Gemini 1.5 Pro | 深度命盤分析、跨古籍綜合分析 | 128K | 高 | 精確度最高、功能完整 |\n| Gemini 1.5 Flash | 基礎星曜解釋、UI 文字潤色 | 128K | 低 | 成本優化、速度快 |\n\n### 上下文快取實現\n\n系統預加載《紫微斗數全書》、《周易》原文等靜態知識庫至 Google 伺服器，解析時僅傳送用戶命盤數據。這一機制可降低 **80% 的重複 Token 費用**。\n\n### 多模態輸入支援\n\n用戶可直接上傳命盤截圖，系統利用 Gemini 1.5 Pro 的視覺識別能力進行 OCR 與邏輯識別，自動提取星曜、宮位、四化等信息。\n\n### 結構化 JSON 輸出\n\n所有分析結果均以結構化 JSON 格式返回，確保前端 UI 與 AI 回應完全同步。當 AI 識別出「化忌」在夫妻宮時，UI 自動觸發紅色高亮效果。\n\n## 文件結構\n\n```\nclient/src/lib/\n├── geminiService.ts              # LLMProvider 類別與多模型編排\n├── 重試機制\n\n系統自動實現指數退避重試：\n\n```\n第 1 次嘗試：立即\n第 2 次嘗試：延遲 1000ms\n第 3 次嘗試：延遲 2000ms\n第 4 次嘗試：延遲 4000ms（最大 10000ms）\n```\n\n### 常見錯誤\n\n| 錯誤 | 原因 | 解決方案 |\n|------|------|----------|\n| API 限流 | 請求過於頻繁 | 自動重試，延遲後重新發送 |\n| 超時 | 網絡問題 | 自動重試，最多 3 次 |\n| 無效 JSON | 模型回應格式錯誤 | 驗證 Schema，記錄錯誤 |\n| 隱私檢測失敗 | 敏感信息未脫敏 | 檢查隱私過濾邏輯 |\n\n## 性能優化\n\n### Token 消耗優化\n\n1. **知識庫快取** - 減少 80% 重複 Token\n2. **JSON 格式** - API Key 正確性\n2. 檢查網絡連接\n3. 確認 API 配額未超出\n4. 查看 Google Cloud 控制台的 API 狀態\n\n### Token 消耗過高\n\n**症狀**：每次分析消耗超過 8000 tokens\n\n**解決方案**：\n1. 驗證知識庫快取是否正常加載\n2. 檢查 Prompt 是否過長\n3. 考慮使用 Flash 模型進行簡單任務\n4. 啟用快取機制\n\n### 分析結果不完整\n\n**症狀**：JSON 回應缺少某些字段\n\n**解決方案**：\n1. 檢查 System Instruction 是否完整\n2. 驗證 JSON Schema 定義\n3. 增加 max_tokens 限制\n4. 查看 API 回應的完整日誌\n\n###
